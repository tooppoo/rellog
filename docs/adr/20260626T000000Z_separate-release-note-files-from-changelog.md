# Separate pending entries, release-note files, and CHANGELOG.md

- Status: Proposed
- Created: 2026-06-26T00:00:00Z

## Context

`rellog` needs to manage several related but distinct artifacts:

- pending changelog entries;
- prepared per-release notes;
- the cumulative `CHANGELOG.md`.

If all information is stored only in `CHANGELOG.md`, release-note preparation has no durable per-release artifact. If pending entries are treated as the final release notes, there is no clear preparation step. If release notes are equated with GitHub Release Notes, `rellog` becomes coupled to GitHub publishing workflows.

The design also needs a stable aggregation order for entries. Allowing arbitrary user-specified entry filenames would make ordering depend on naming choices and could make release-note generation confusing.

This decision should be recorded as an ADR because it defines the repository file model and the lifecycle of `rellog` artifacts.

## Decision

`rellog` will use the following repository layout:

```text
.rellog/
  config.kdl
  entries/
    0001.md
    0002.md
  release-notes/
    <release-id>.md
CHANGELOG.md
```

`.rellog/entries/` stores pending changelog entries for the next release-note preparation.

`.rellog/release-notes/` stores prepared plain Markdown release-note files by release id.

`CHANGELOG.md` stores the cumulative release record.

Release-note files are plain Markdown artifacts. They are not GitHub Release Notes. `rellog` does not create GitHub Releases, but other tooling may reuse these files.

Entry filenames are generated by `rellog` and cannot be specified by users. The intended format is a zero-padded sequence number such as:

```text
0001.md
0002.md
0003.md
```

Entries are aggregated in filename order. If a project must reorder entries before preparation, it can manually renumber the files.

For v0, release ids should be path-safe because they are used as filenames under `.rellog/release-notes/`. A conservative allowed form is:

```text
[A-Za-z0-9._-]+
```

## Alternatives Considered

### Store everything only in CHANGELOG.md

This was not selected because it does not provide a durable per-release Markdown artifact. A separate release-note file is useful for review, reuse by external release tooling, and workflow guards.

### Store release notes only in GitHub Releases

This was not selected because `rellog` should not be coupled to GitHub Release publishing. The release-note file is a repository-managed Markdown artifact, not a GitHub Release Note.

### Allow arbitrary entry filenames

This was not selected because arbitrary names can make aggregation order unclear. Sequential filenames provide a simple and predictable ordering rule.

### Use timestamp-based filenames

This was not selected for v0 because timestamps are harder to manually reorder. Sequential filenames are simpler and still allow manual reordering by renumbering.

## Consequences

### Positive Consequences

- Pending entries, prepared release notes, and cumulative changelog output are clearly separated.
- Per-release Markdown files can be reviewed and reused independently of `CHANGELOG.md`.
- Entry aggregation order is deterministic.
- The file model remains independent from GitHub Release publishing.

### Negative Consequences

- Manual reordering requires renumbering files.
- Release ids must be constrained or encoded because they are used as filenames.
- The repository contains both per-release files and a cumulative changelog, which introduces some duplication.

### Neutral Consequences

- The exact `config.kdl` schema remains a future implementation detail.
- Other release tooling may consume `.rellog/release-notes/<release-id>.md`, but that is outside `rellog`'s core responsibility.
