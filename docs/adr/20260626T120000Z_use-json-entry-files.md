# Use JSON files for pending entries

- Status: Accepted
- Created: 2026-06-26T12:00:00Z
- Updated: 2026-06-27T12:05:58Z
- Amended by: [Pivot to a VCS-independent links reference model](20260627T120558Z_pivot-to-vcs-independent-links-model.md)

## Context

`rellog` pending entries were originally described as frontmatter Markdown files with sequence filenames such as `0001.md`.

That format is friendly to read directly, but it makes validation depend on frontmatter parsing and creates ambiguity around metadata types. The project also needs deterministic entry files in e2e tests while moving to timestamp-generated filenames.

This decision should be recorded as an ADR because entry files are repository-managed state and are the primary input to release-note preparation.

## Decision

`rellog` will store each pending entry as one JSON file under `.rellog/entries/`.

Entry filenames are generated by `rellog` from UTC timestamps with nanosecond precision:

```text
20260626T123456.123456789Z.json
```

Normal entries use this shape:

```json
{
  "kind": "changed",
  "targets": ["rellog"],
  "links": [],
  "body": "Entry body"
}
```

Empty entries use the same shape, with empty arrays:

```json
{
  "kind": "empty",
  "targets": [],
  "links": [],
  "body": "No changelog-worthy changes."
}
```

The `targets` and `links` keys are required. They must be arrays, and empty values must be represented as `[]`.

When `kind` is `empty`, `targets` and `links` must both be `[]`.

For deterministic e2e tests, `rellog add` and `rellog add-empty` may accept a hidden `--debug-datetime <timestamp>` option. This is a test hook, not a user-facing command option, and it must not appear in normal help output.

v0 does not support entry reordering. Entries are aggregated by filename order.

## Alternatives Considered

### Frontmatter Markdown files

This was not selected because frontmatter parsing increases validation complexity and makes metadata type errors less direct. Markdown remains the generated release-note format instead.

### Sequence-numbered JSON files

This was not selected because sequence filenames are more likely to conflict when multiple contributors add entries concurrently.

### Single `entry.json`

This was not selected because one shared JSON document would create avoidable merge conflicts and make independent entry additions harder to review.

### Single `entry.jsonl`

This was not selected because JSONL is append-friendly but awkward for deletion, reordering, and precise per-entry validation messages.

## Consequences

### Positive Consequences

- Entry parsing and metadata validation become simpler and more reliable.
- Required array fields make the entry shape stable for tools.
- Timestamp filenames reduce ordinary merge conflicts between concurrently added entries.
- Human review focuses on `rellog status` and `rellog prepare` preview, which show the release-facing content.

### Negative Consequences

- Pending entry files are less pleasant to read directly than Markdown.
- Manual reordering is omitted in v0.
- Tests need a hidden timestamp hook to avoid nondeterministic filenames.

### Neutral Consequences

- Release-note files remain Markdown.
- Existing frontmatter Markdown entry files are not part of the v0 compatibility contract.
