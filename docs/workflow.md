# rellog workflow

This document describes the intended CHANGELOG and release-note-file workflow for `rellog`.

It focuses on project state transitions. See [commands.md](commands.md) for individual command behavior.

## Principle

`rellog` separates development history from release explanation.

Git history records how work was divided and performed. A CHANGELOG records what changed in a release and how that change should be communicated to users, operators, or maintainers.

`rellog` therefore uses explicit changelog entries as its primary input. It does not reconstruct final release notes from commit history.

## Repository layout

A typical repository using `rellog` has the following files:

```text
.rellog/
  config.kdl
  entries/
    0001.md
    0002.md
  release-notes/
    <release-id>.md
CHANGELOG.md
```

The important distinction is that pending changelog entries, prepared release-note files, and the cumulative `CHANGELOG.md` are separate artifacts.

- `.rellog/entries/` stores pending changelog entries for the next release preparation.
- `.rellog/release-notes/` stores prepared plain Markdown release-note files by release id.
- `CHANGELOG.md` stores the cumulative changelog assembled from prepared release notes.

## Artifact roles

### Pending entries

Pending entries are the source records for the next release explanation.

They are not generated from Git history. They are explicit records written at release-note granularity.

Entry filenames are generated by `rellog` as sequence numbers such as `0001.md`, `0002.md`, and `0003.md`. Users do not choose entry filenames. The filename order is the stable aggregation order.

If a project must reorder entries before preparation, it can still do so by manually renumbering files. This keeps the rule simple while avoiding arbitrary filename-based ordering.

Pending entries are not expected to exist at all times. In ordinary development, `.rellog/entries/` may often be empty. That is acceptable because `rellog` is not designed to force a perfectly maintained change record during day-to-day work.

### Empty entry

An empty entry explicitly records that there is nothing changelog-worthy for the next release.

This is not a validation bypass option. It is a repository state.

A pending entry state may be either:

- normal: one or more normal entries exist;
- empty: one empty entry exists;
- invalid: normal entries and an empty entry coexist, or entries are malformed;
- absent: no pending entries exist.

Only `normal` and `empty` can be consumed by release-note preparation. `absent` is allowed during ordinary development, but `rellog prepare <release-id>` fails in that state.

### Release-note files

`rellog` release notes are plain Markdown files under `.rellog/release-notes/`.

They are not GitHub Release Notes, and `rellog` does not create GitHub Releases. Other release tooling may reuse `.rellog/release-notes/<release-id>.md`, but that is outside `rellog`'s core responsibility.

### CHANGELOG.md

`CHANGELOG.md` is the cumulative release record. `rellog prepare <release-id>` appends the prepared release-note content to `CHANGELOG.md`.

## Project states

`rellog` models the project as moving through the following documentation states.

```text
No pending entry
  -> Pending normal entries
  -> Prepared release note
  -> Published or consumed by external release workflow

No pending entry
  -> Pending empty entry
  -> Prepared empty release note
  -> Published or consumed by external release workflow
```

The key v0 rule is:

```text
release publishing requires a prepared release-note file
```

Release-note preparation itself still needs entries to consume. If no pending entry exists at preparation time, the user can either add normal entries or add an explicit empty entry.

## State transitions

### Initial state

After initialization, the repository has `.rellog/`, `.rellog/entries/`, `.rellog/release-notes/`, and usually `CHANGELOG.md`.

At this point, there are no pending entries.

```text
.rellog/entries/        empty
.rellog/release-notes/  may be empty or contain older release notes
CHANGELOG.md            may be empty or contain older releases
```

This state is valid for ordinary development. It only becomes a problem when the project tries to prepare release notes without first creating a normal entry or an explicit empty entry.

### Pending normal entries

When a change should be mentioned in the next release, a normal entry is added.

```text
.rellog/entries/
  0001.md
  0002.md
```

This state means that the next release has changelog-worthy changes. The entries are ordered by filename and later aggregated in that order.

### Pending empty entry

When there is nothing changelog-worthy for the next release, an empty entry is added.

```text
.rellog/entries/
  0001.md
```

The entry body represents an explicit decision such as:

```text
No changelog-worthy changes.
```

This state means that the project intentionally records an empty release note. It is not the same as forgetting to write entries.

### Invalid pending state

A repository is in an invalid pending state when pending entries contradict each other or cannot be parsed.

Examples:

- an empty entry coexists with normal entries;
- an entry has invalid metadata;
- an entry body is empty;
- an entry has an unknown kind or target, depending on project policy.

Release preparation should fail in this state.

### Prepared release note

During release preparation, pending entries are consumed into a release-note file.

```text
.rellog/release-notes/v1.0.1.md
```

The same release-note content is appended to `CHANGELOG.md`, and consumed files are removed from `.rellog/entries/`.

After preparation:

```text
.rellog/entries/                    empty
.rellog/release-notes/v1.0.1.md     exists
CHANGELOG.md                        includes v1.0.1 section
```

### Release workflow handoff

After a release-note file exists, later release workflow steps may use it as an input.

For example, external tooling may:

- create a GitHub Release using the Markdown file;
- include the Markdown file in a package publishing step;
- copy the file into website documentation;
- verify that release notes exist before publishing artifacts.

Those steps are outside `rellog`'s core responsibility.

## Release preparation sequence

A typical release preparation sequence is:

```text
1. Add normal entries, or add an explicit empty entry when there is nothing to mention.
2. Validate pending rellog state.
3. Provide a release id from the external release process.
4. Aggregate entries in filename order.
5. Create .rellog/release-notes/<release-id>.md.
6. Append the same release-note section to CHANGELOG.md.
7. Delete consumed pending entries.
8. Open a pull request containing only rellog-managed documentation changes.
```

If step 1 is skipped, release preparation fails. This failure is the reminder point: rellog does not require perfect entry maintenance throughout ordinary development, but it does require an explicit record before release notes are prepared.

The resulting pull request should contain only rellog-managed documentation artifacts:

- create `.rellog/release-notes/<release-id>.md`;
- update `CHANGELOG.md`;
- delete consumed files from `.rellog/entries/`.

It should not update versions, update package manifests, create tags, create GitHub Releases, or publish artifacts.

## GitHub Actions workflow role

`rellog` should provide a GitHub Action for a changesets-like experience without version management.

In v0, the primary workflow gate is the release-note gate:

```text
release-note gate:
  require .rellog/release-notes/<release-id>.md before publish-oriented workflow steps
```

The action layer should also be able to create a pull request for release preparation changes.

The important point is that the action does not infer final release notes from Git history. It only checks and consumes explicit rellog entries when preparing release notes, and later verifies that a release-note file exists before release-oriented workflow steps proceed.

## Future AI-assisted workflow

AI support may be considered after the core workflow is stable.

The acceptable role of AI is to suggest candidate changelog entries from issues, pull requests, diffs, or commit history. The final changelog entry should still be reviewed and accepted as an explicit project record.

In other words, AI may help draft entries, but `rellog` should not silently infer final release notes from Git history.
